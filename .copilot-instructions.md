## The 11 Commandments (Your North Star)

**ALWAYS CHECK THESE BEFORE SUGGESTING ANY FEATURE:**

1. **Helpful Neighbor** - Anticipate needs, link to KB/training, educate
2. **Free as Possible** - Local features free forever; only charge if server costs exist
3. **Register Not Pay** - Generous free tier; ask for registration, not money
4. **Advice Not Sales** - Educational copy; drive to KB articles, not upsells
5. **Drive to KB** - Every diagnostic/treatment links to free knowledge base
6. **Drive to Training** - Every diagnostic/treatment links to free training video
7. **Ridiculously Good** - Better than premium alternatives, free
8. **Inspire Confidence** - UX so intuitive users assume all WordPress works this easily
9. **Show Value (KPIs)** - Track time saved, issues fixed, value delivered
10. **Beyond Pure (Privacy)** - Consent-first, transparent, no dark patterns
11. **Talk-Worthy** - So good people share, recommend, celebrate

**Test Every Feature Against This:**
```
GREEN LIGHT - Build it:
✅ Makes WordPress management easier for everyone
✅ Can be done well in free tier
✅ Creates educational opportunity
✅ Delivers measurable KPI improvement
✅ Builds confidence/trust
✅ Users would share it

YELLOW LIGHT - Caution:
⚠️ Requires infrastructure (must have generous free tier)
⚠️ Complex UI (must be exceptionally intuitive)
⚠️ Collects data (must be consent-first, transparent)
⚠️ Could feel sales-y (rework with "helpful neighbor" lens)

RED LIGHT - Don't build:
❌ Creates dependency instead of empowerment
❌ Requires payment for basic functionality
❌ Hides information behind upsells
❌ Feels manipulative/sales-driven
❌ Collects data without clear user benefit
❌ "Enterprise only" or "Contact sales"
```

## Code Quality Standards

### Security (Non-Negotiable)
- ✅ Nonce verify on AJAX/forms: `check_ajax_referer()`
- ✅ Capability check: `manage_options` or `manage_network_options`
- ✅ Input sanitize: `sanitize_text_field()`, `sanitize_key()`, etc.
- ✅ Output escape: `esc_html()`, `esc_attr()`, `esc_url()`, `wp_kses_post()`
- ✅ No direct SQL: Use `$wpdb->prepare()` if needed
- ✅ No `eval()` or dynamic code execution

### DRY Architecture (Enforce Always)
- Base classes: `Treatment_Base`, `AJAX_Handler_Base`, `Diagnostic_Base`
- All treatments extend `Treatment_Base` (43/43 ✅)
- All AJAX handlers extend `AJAX_Handler_Base` (17/25 ✅)
- Use registries: `Diagnostic_Registry`, `Treatment_Registry`
- No duplicate code across files

### Type Safety
- `declare(strict_types=1);` in all PHP files
- Use return type hints: `: bool`, `: array`, `: string`
- Use parameter type hints: `function run(string $id): bool`
- WordPress type stubs supported by Intelephense

### Philosophy Integration (Every Feature)
- Links to KB: Every diagnostic/treatment must link to knowledge base article
- Links to training: Every diagnostic/treatment must link to training video
- KPI tracking: Use `KPI_Tracker::record_treatment_applied(__CLASS__, $minutes_saved);`
- Plain English: No jargon; explain why it matters
- Consent-first: Any data collection needs opt-in
- Multisite aware: Check `is_multisite()`, use `manage_network_options` when needed

## When Reviewing Code: Your Checklist

**Before approving changes, ask:**

1. ✅ **Philosophy**: Does this align with the 11 commandments?
2. ✅ **DRY**: Does it reuse base classes? No duplicate code?
3. ✅ **Security**: Nonce? Capability? Sanitize? Escape?
4. ✅ **Education**: KB link? Training link? Clear explanation?
5. ✅ **KPI**: Does it track value delivered (time saved, issues fixed)?
6. ✅ **Reversibility**: If it's a treatment, can it be undone?
7. ✅ **Multisite**: Does it handle single-site and network contexts?
8. ✅ **Type Safety**: Are type hints used? strict_types declared?
9. ✅ **Plain English**: Would a non-technical user understand the message?
10. ✅ **Tests**: Does it have real-world validation?

**If ANY answer is ❌, ask follow-up questions before proceeding.**

## Common Anti-Patterns (Stop These Immediately)

### ❌ Philosophy Violations
```php
// WRONG: Paywall on local feature
if (!is_pro_user()) {
    return ['error' => 'Upgrade to Pro'];
}

// RIGHT: Free locally, optional registration for cloud features
// No registration required = full local diagnostics
// Registration optional = adds cloud sync
```

### ❌ Security Issues
```php
// WRONG: Direct $_POST without sanitization
$value = $_POST['setting'];

// RIGHT: Sanitize + escape
$value = sanitize_text_field($_POST['setting'] ?? '');
```

### ❌ DRY Violations
```php
// WRONG: Duplicate nonce check in 3 AJAX handlers
if (!wp_verify_nonce($_POST['nonce'], 'action')) { ... }

// RIGHT: Use AJAX_Handler_Base which handles this centrally
class My_Handler extends AJAX_Handler_Base {
    public static function handle(): void {
        self::verify_request('nonce_action', 'manage_options');
        // Already verified
    }
}
```

### ❌ Missing Education
```php
// WRONG: No KB/training links
__('Your site is not optimized', 'wpshadow')

// RIGHT: Links to learning resources
sprintf(
    __('Your site could be 30%% faster. <a href="%s">Learn why this matters</a>', 'wpshadow'),
    'https://wpshadow.com/kb/performance-optimization'
)
```

## Your Decision Framework

### When User Asks for a Feature
1. **Check Philosophy**: Does it pass the 11 commandments test?
2. **Propose Plan**: Show how it aligns with philosophy
3. **Ask Confirmation**: "Ready to build?" (don't assume)
4. **Implement**: Follow code standards above
5. **Test**: Verify in WordPress, check pre-commit hooks

### When You Spot a Problem
1. **Don't Fix Silently**: Flag it to user first
2. **Explain Why**: Reference relevant commandment or code standard
3. **Propose Fix**: Specific, actionable recommendation
4. **Get Approval**: Let user decide
5. **Execute**: Implement with tests

### When Philosophy vs. Feature Request Conflict
**Philosophy Always Wins.** Explain gently:
> "This feature would require users to upgrade to Pro for basic diagnostics. That violates Commandment #2 (Free as Possible). Alternative: Make core diagnostic free, premium features are just 'convenience cloud features' they can opt into."
