# Contributing to This Is My URL Shadow

Thank you for helping improve This Is My URL Shadow.

We want contributions to feel like they came from a **helpful neighbor**: clear, practical, kind, and safe.

---

## Before You Start

Please read these first:

1. [`docs/CORE_PHILOSOPHY.md`](docs/CORE_PHILOSOPHY.md)
2. [`docs/FEATURES.md`](docs/FEATURES.md)
3. [`SECURITY.md`](SECURITY.md)
4. [`SUPPORT.md`](SUPPORT.md)

If your change conflicts with the philosophy, the philosophy wins.

---

## What We Welcome

We welcome contributions that make This Is My URL Shadow:

- easier to understand
- safer to use
- more accessible
- more reliable
- more inclusive
- more useful for real WordPress site owners

### Great Contribution Types

- Bug fixes
- Accessibility improvements
- Documentation improvements
- Clearer UI copy
- Diagnostic accuracy improvements
- Safer treatment behavior
- Better onboarding and support materials
- Tests and validation improvements

---

## What Good Contributions Look Like

A strong contribution should usually do one or more of these:

- explain **why** something matters, not just **what** changed
- reduce confusion for site owners
- avoid fear-based or salesy wording
- preserve privacy and consent
- work well with keyboards, screen readers, and smaller screens
- keep risk low and behavior predictable

---

## Non-Negotiable Standards

### 1. Helpful Neighbor Tone
No manipulative copy. No scare tactics. No nagware.

### 2. Accessibility First
If a feature is not accessible, it is not complete.

### 3. Safe by Default
Risky actions should be reversible, explained, and clearly confirmed.

### 4. Privacy First
Do not add hidden tracking, silent third-party calls, or confusing consent behavior.

### 5. Plain English
Assume the reader is intelligent but busy. Use clear language instead of jargon whenever possible.

## Development Workflow

### Local Setup
This Is My URL Shadow is intended to be worked on in:

- GitHub Codespaces, or
- VS Code with Dev Containers

Basic flow:

```bash
git clone https://github.com/thisismyurl/thisismyurl-shadow.git
cd thisismyurl-shadow
```

Then open the project in the configured development environment.

### Branch Naming
Suggested patterns:

- `fix/...`
- `feature/...`
- `docs/...`
- `chore/...`

Examples:

- `fix/dashboard-diagnostic-toggle`
- `docs/support-policy`
- `feature/new-accessibility-diagnostic`

## Pull Request Expectations

A pull request should be small enough to review clearly.

Please include:

- a short explanation of the problem
- what changed
- why the change is the right fix
- any screenshots for UI changes
- any testing or verification notes

### PR Checklist

Before opening a PR, confirm:

- [ ] I read the core philosophy
- [ ] My change is written in a helpful, plain-English tone
- [ ] I considered accessibility and keyboard/screen-reader use
- [ ] I avoided manipulative or sales-driven copy
- [ ] I did not add unnecessary tracking or privacy risk
- [ ] I tested the behavior I changed
- [ ] I updated docs if user-facing behavior changed

### Accessibility Review Checklist

For any user-facing change, try to check:

- Can this be used without a mouse?
- Does the label or instruction still make sense when read aloud by a screen reader?
- Is the wording understandable for a busy or non-technical site owner?
- Does the UI avoid unnecessary motion, surprise, or panic-inducing language?
- Are errors or warnings explained clearly enough for someone under stress?
- Would this still feel usable to someone with low vision, cognitive load, or motor limitations?

If the answer is "not yet," the change likely needs another pass.

## Documentation Contributions

Documentation is a first-class contribution.

Good docs should:

- answer the real question a user is likely asking
- avoid unexplained jargon
- separate **current reality** from **future plans**
- be honest about what is shipped vs experimental
- help both beginners and experienced users

---

## Bug Reports and Feature Requests

Please use the repository templates when available.

If you are reporting a bug, include:

- WordPress version
- PHP version
- plugin version
- exact steps to reproduce
- what you expected
- what actually happened
- screenshots or logs if relevant

If you are proposing a feature, explain:

- the user problem
- why it matters
- how it aligns with This Is My URL Shadow philosophy
- any accessibility or privacy considerations

---
